## 一、核心洞察（Insight）

### 1.1 真实痛点

结合背景文档 `context/ShipSage报价配置现状.md`、`docs/ShipSage_Billing_系统文档.md` 以及现有系统页面能力，当前报价体系存在四个核心问题：

1. **标准版报价缺少可视化配置入口**
   当前系统理论上存在 Template 标准版与 Company 客户版两套报价体系，但运营侧实际可操作页面主要围绕客户级配置展开，标准版更像底层基准数据，而不是可管理的产品对象。
2. **客户报价维护方式偏“导入驱动”，不适合高频调价**
   当前主流程依赖上传报价文件、创建草稿版本、保存生效，缺少面向运营的明细行级编辑能力，导致小范围调价、核价、快速修正的成本较高。
3. **客户侧难以看到“完整可读”的最终报价**
   现有计费逻辑以标准版为底稿、客户版做差异覆盖，正式生效价是运行时叠加结果。内部尚可理解，但客户侧或 OMS 很难直接展示一份“当前对我生效的完整报价”。
4. **查询能力存在，但与业务目标不匹配**
   系统已有“定制报价查询”能力，但当前查询结果更接近后台数据清单，缺少版本、生效区间、来源口径和客户可读视图，难以支撑客户自助查询和内部核价场景。

### 1.2 本质问题判断

这不是单纯的页面优化需求，而是一个**报价产品化能力重构**需求：

- 把“标准版 / 客户版 / 最终生效价”从隐性数据关系，转成显性的产品概念
- 把“导入式维护”升级为“页面可视化维护 + 版本化发布”
- 把“后台数据查询”升级为“内部核对 + 客户展示”的双端查询能力

### 1.3 业务价值评估

| 维度    | 评估         | 说明                                |
| ----- | ---------- | --------------------------------- |
| 运营效率  | **High**   | 小范围调价、核价、版本排期不再依赖 Excel 反复导入      |
| 客户体验  | **High**   | 客户可直接查看当前、未来、历史报价，减少客服解释成本        |
| 数据一致性 | **High**   | 统一“标准版、客户覆盖、最终生效价”口径，降低展示与计费不一致风险 |
| 交付风险  | **Medium** | 需兼容现有计费链路和版本逻辑，不能推翻底层费率模型         |

### 1.4 现有系统边界

- 现有系统已存在：
  - 公司报价列表/详情
  - 服务报价版本管理
  - 客户定制报价查询
  - 草稿版本、折扣复制、复制到其他客户等流程
- 现有系统未成熟具备：
  - 标准版独立管理视图
  - 面向运营的行级直接编辑
  - 面向客户的最终报价查询页
  - “完整报价快照”式展示口径

## 二、用户故事（User Stories）

### US-01 标准版报价运营维护

```text
As 计费运营/产品运营
I want to 在后台直接查看并编辑标准版报价
So that 我可以把标准价作为独立资产管理，而不是只能通过底层导入或隐式数据维护
```

典型场景：

- 运营按费用维度查看标准版报价列表
- 进入服务详情后，查看某一版本的标准价明细
- 直接修改行级价格并保存为草稿或直接生效
- 版本在未来某日生效，保留历史记录可追溯

### US-02 客户版报价运营维护

```text
As 计费运营/客服
I want to 在客户报价页面按客户、服务、费用维度编辑覆盖价格
So that 我能快速响应客户调价需求，而不必每次都走完整导入流程
```

典型场景：

- 选定某客户后查看其当前报价配置
- 清晰识别哪些服务使用标准价，哪些存在客户覆盖
- 对已有明细项做价格级编辑，不改结构
- 支持草稿保存、发布生效、直接生效、复制到其他客户

### US-03 内部最终报价查询

```text
As 运营/客服/销售
I want to 查询任意客户在任意日期的最终生效报价
So that 我可以对外答复客户、内部核价，并解释版本和生效时间
```

典型场景：

- 输入客户、日期、费用维度、服务后查看最终价格
- 可见版本名称、生效区间、来源类型
- 支持当前、未来、历史查询
- 支持导出结果用于对账或沟通

### US-04 客户侧报价自助查询

```text
As OMS 客户
I want to 在系统里直接查看当前和历史对我生效的报价
So that 我可以自助确认收费标准，而不必依赖人工发送报价文件
```

典型场景：

- 客户进入报价查询页
- 仅看到对自己生效的最终价格，不暴露标准价和内部覆盖逻辑
- 可以查看当前报价、未来排期报价、历史报价
- 可以按服务/费用类型检索并导出

## 四、验收标准（Acceptance Criteria）

### AC-01 标准版报价管理

- [ ] 后台存在独立的标准版配置入口
- [ ] 可按费用维度、服务、版本、生效时间查看标准版报价
- [ ] 可在页面内对既有明细行做价格编辑
- [ ] 支持草稿保存和发布生效
- [ ] 历史版本可查询，不被覆盖丢失

### AC-02 客户版报价管理

- [ ] 可按客户、费用维度、服务查看客户报价
- [ ] 页面中能明确识别“沿用标准版”与“存在客户覆盖”
- [ ] 支持对既有明细行做价格编辑
- [ ] 支持两种生效方式：草稿后发布、直接保存生效
- [ ] 保留现有导入、折扣、复制、历史查看能力

### AC-03 内部最终报价查询

- [ ] 内部用户可查询任意客户在指定日期的最终生效价
- [ ] 查询结果包含服务、费用维度、最终价格、生效区间、来源说明
- [ ] 支持当前、未来、历史三个时间维度
- [ ] 支持按客户、日期、服务、费用维度筛选
- [ ] 支持导出

### AC-04 客户报价查询

- [ ] 客户在 OMS 侧可查看对自己生效的最终报价
- [ ] 不展示标准价、覆盖差异、内部来源细节
- [ ] 支持当前、未来、历史报价查看
- [ ] 支持按服务/费用类型筛选
- [ ] 查询结果与内部最终报价口径一致

### AC-05 一致性与版本能力

- [ ] 客户未覆盖项时，最终价正确继承标准版
- [ ] 客户已覆盖项时，最终价正确显示客户价
- [ ] 发布或直接生效后，版本区间不冲突
- [ ] 旧版本保留可追溯，不发生无痕覆盖

## 六、建议方案（Proposed Solution）

### 6.1 方案方向

采用“**配置中心统一、查询口径统一、底层逻辑不推翻**”的方案：

1. **配置中心统一**
   内部报价配置统一进入一个中心，但分为“标准版配置”和“客户版配置”两个视图，避免继续将标准版隐藏在底层模型里。
2. **查询口径统一**
   内部与客户共用“最终生效价”口径；客户只看最终结果，内部可见更多上下文信息。
3. **维护方式双轨**
   页面直编与导入并存，满足高频小改与批量变更两类场景。
4. **版本化治理继续保留**
   不做无痕覆盖，继续保留草稿、发布、生效时间、历史追溯能力。

### 6.2 推荐交互原则

| 设计原则    | 说明                  |
| ------- | ------------------- |
| 标准版显性化  | 标准版必须作为独立配置对象出现     |
| 客户侧结果导向 | 客户只关心“我现在生效的价格”     |
| 内部侧解释能力 | 内部需要能解释版本、生效区间、来源口径 |
| 小改快、批改稳 | 页面直编适合小改，导入适合批量维护   |
