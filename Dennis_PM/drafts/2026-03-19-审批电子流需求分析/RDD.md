## 一、核心洞察 (Insight)

### 1.1 真实痛点

仓盛内部存在多类高频业务审批（约车付款、开票申请、合同生效、渠道充值），目前**全部依赖外部第三方App（如微信审批、钉钉等）流转**，导致：

1. **系统无记录**：审批结果无法与 ShipSage 业务数据关联，财务对账、合规审计困难
2. **联动断层**：约车账单生成后需人工跨系统发起付款，客户开票需邮件往来，效率低下
3. **权限散乱**：审批人由人工指定，无法基于角色动态路由，临时代理无法追溯
4. **客户体验差**：客户无法在 OMS 内直接发起开票申请和查看发票，需要通过客服中转

### 1.2 业务价值评估

| 维度 | 评估 | 说明 |
|------|------|------|
| 效率提升 | **High** | 约车付款从账单→申请→审批全链路系统化，减少人工操作 |
| 合规/审计 | **High** | 所有审批留存完整日志，满足财务审计要求 |
| 客户体验 | **Medium** | 客户可在 OMS 自助发起开票申请，自助下载发票 |
| 风险控制 | **Medium** | 金额/角色双重校验，避免越权操作 |

### 1.3 现有系统边界

- Admin 财务模块已有"审批"入口（库存调整/盘点调整），**仅限库存类审批**，本次为扩展
- 约车模块已存在（Admin → 服务 → 预约），可联动
- 客户充值列表已存在（OMS → 计费 → 付款），可作为开票金额数据源
- 系统已有角色管理（Admin → 设置 → 角色管理），审批人配置可复用

## 二、用户故事 (User Stories)

### US-01 约车付款申请（内部客服发起）

```
As 内部客服
I want to 在约车账单生成后直接点击触发付款申请
So that 审批通过后财务可快速执行付款，全程有系统记录
```

**详细场景**：
- 触发点：约车模块账单出具后，账单详情页出现「发起付款申请」按钮
- 申请内容：金额（0-$1,000）、约车单号、收款方、付款说明
- 审批层级：两级（如：主管审批 → 财务审批）
- 审批通过后：付款申请状态更新为「待付款」，财务人工执行付款

### US-02 客户付款开票申请（客户在 OMS 发起）

```
As 客户
I want to 在充值/付款完成后在 OMS 内发起开票申请
So that 财务审核后上传发票，我可以在 OMS 直接下载
```

**详细场景**：
- 触发点：OMS 充值模块 → 选择已付款记录 → 「申请开票」
- 申请内容：开票金额（从充值记录获取）、发票类型（增值税专票/普票）、客户录入的开票信息（抬头、税号、地址、电话、开户行）
- 审批层级：可配置（默认：财务角色审批）
- 审批通过后：财务上传发票文件，客户在「我的发票」查看并下载

### US-03 客户合同生效审批（内部发起）

```
As 销售/运营
I want to 在系统内发起合同生效申请（新签/续签）
So that 审批记录留存系统，合同生效有据可查
```

**详细场景**：
- 触发点：Admin → 客户管理 → 合同管理 → 「发起审批」
- 申请内容：客户名称、合同类型（新签/续签）、合同有效期、上传合同附件
- 审批通过后：合同状态变更为「已生效」，**暂不触发其他系统动作**

### US-04 渠道账号充值审批（内部发起）

```
As 运营
I want to 在系统内发起物流渠道账号充值申请
So that 审批留存记录，财务执行有据可查
```

**详细场景**：
- 触发点：Admin 计费 → 物流管理 → 渠道列表 → 「发起充值申请」
- 申请内容：渠道名称、充值金额、充值原因
- 审批通过后：状态更新为「待执行」，**人工充值，仅做记录**

### US-05 审批委托（代理审批）

```
As 审批人（如因出差/休假）
I want to 将我的审批权限临时委托给其他用户
So that 业务审批不因我不在而中断
```

## 三、功能范围 (Scope)

### 3.1 核心功能模块

| 模块 | 功能点 | 优先级 |
|------|--------|--------|
| **审批流引擎** | 可配置审批节点、条件分支、审批角色 | P0 |
| **审批流配置后台** | 管理员配置各审批类型的流程模板 | P0 |
| **约车付款申请** | 从约车账单触发，两级审批，付款状态联动 | P0 |
| **客户开票申请** | 客户OMS端发起，财务上传发票，客户下载 | P0 |
| **合同生效审批** | 内部发起，附件上传，仅做记录 | P1 |
| **渠道充值审批** | 内部发起，仅做记录 | P1 |
| **审批委托** | 设置委托人、委托时间范围 | P1 |
| **审批消息通知** | 站内消息 + 邮件通知（待审批/已通过/已拒绝） | P1 |
| **审批记录查询** | 申请人/审批人均可查看历史记录和当前状态 | P0 |

### 3.2 MVP 范围（建议）

**MVP（优先交付）**：约车付款申请 + 客户开票申请 + 基础审批引擎（固定两级）+ 审批记录

**Phase 2**：合同生效审批 + 渠道充值审批 + 审批委托 + 可配置流程模板

> **理由**：约车付款和开票申请是日常高频业务，且有系统联动价值；合同和充值审批仅做记录，优先级略低。

## 四、验收标准 (Acceptance Criteria)

### AC-01 审批流配置
- [ ] 管理员可在后台为每种审批类型配置审批节点（节点名称、审批角色、通过条件）
- [ ] 支持配置条件分支（如：金额 > $500 时需额外审批节点）
- [ ] 审批流模板变更后，存量进行中的申请继续走原流程，新申请走新流程

### AC-02 约车付款申请
- [ ] 约车账单生成后，账单详情出现「发起付款申请」按钮，点击预填约车单信息
- [ ] 申请金额上限 $1,000 校验，超出时提示错误
- [ ] 两级审批人依次收到站内通知
- [ ] 第一级拒绝后流程终止，申请人收到通知和拒绝原因
- [ ] 全部通过后，约车账单状态更新为「付款审批通过/待付款」

### AC-03 客户开票申请
- [ ] 客户在 OMS 充值模块选择已付款记录后可发起开票申请
- [ ] 客户首次申请需录入开票信息（抬头、税号等），后续可复用历史信息
- [ ] 审批通过后，财务可上传发票附件（PDF/OFD格式）
- [ ] 客户在 OMS「我的发票」可查看申请状态和下载发票
- [ ] 发票上传后客户收到站内通知

### AC-04 合同生效审批
- [ ] 支持上传合同附件（PDF，≤20MB）
- [ ] 审批通过后合同状态变更为「已生效」，记录生效时间和审批人

### AC-05 渠道充值审批
- [ ] 审批通过后状态更新为「待执行」
- [ ] 执行人确认充值完成后可更新为「已完成」并备注实际充值金额

### AC-06 审批委托
- [ ] 审批人可设置委托人（仅限同角色或上级角色用户）
- [ ] 委托期间，被委托人可代为审批，审批记录中注明「代理审批」
- [ ] 委托到期自动失效

### AC-07 通用
- [ ] 所有审批操作有完整日志（操作人、时间、操作内容、IP）
- [ ] 申请人随时可查看当前审批状态和各节点处理情况
- [ ] 审批人可在移动端（响应式）处理审批

## 五、非功能性需求 (NFR)

| 类型 | 要求 |
|------|------|
| 性能 | 审批流转通知延迟 ≤ 30秒 |
| 安全 | 申请人只能查看自己的申请；审批人只能看到待自己审批的单据；发票文件需权限控制 |
| 审计 | 所有状态变更有时间戳和操作人记录，不可删除 |
| 兼容 | OMS 客户端审批页面支持响应式（移动端可用） |
| 可用性 | 审批引擎故障不影响其他业务模块正常使用 |

## 六、建议方案 (Proposed Solution)

### 方案选型：方案B — 可配置审批流（已确认）

**核心设计思路**：

```
审批流模板（ApprovalTemplate）
  └── 审批节点（ApprovalNode）× N
        ├── 节点类型：会签 / 或签
        ├── 审批角色：角色ID列表
        └── 条件规则：金额范围、申请类型等

审批申请（ApprovalRequest）
  ├── 业务类型：TRUCK_PAYMENT / INVOICE / CONTRACT / CHANNEL_RECHARGE
  ├── 关联业务ID：约车单ID / 充值记录ID / 合同ID / 渠道ID
  └── 审批实例（ApprovalInstance）× N
        ├── 对应节点
        ├── 审批人
        └── 状态：PENDING / APPROVED / REJECTED
```

**各审批类型系统联动设计**：

| 审批类型 | 触发入口 | 通过后联动 | 联动方式 |
|---------|---------|-----------|--------|
| 约车付款 | 约车账单详情 → 「发起付款申请」 | 账单状态 → 「付款审批通过」 | 系统自动更新 |
| 客户开票 | OMS充值记录 → 「申请开票」 | 财务上传发票 → 客户可下载 | 财务手动上传，系统存储 |
| 合同生效 | Admin客户管理 → 「发起审批」 | 合同状态 → 「已生效」 | 系统自动更新 |
| 渠道充值 | Admin渠道管理 → 「发起充值申请」 | 状态 → 「待执行」 | 人工确认后更新 |

### MVP 交付建议

**Phase 1（4-6周）**：
1. 审批流引擎（可配置节点 + 角色）
2. 约车付款申请（联动约车账单）
3. 客户开票申请（联动充值记录 + 发票上传下载）
4. 基础审批记录和通知

**Phase 2（2-3周）**：
5. 合同生效审批
6. 渠道充值审批
7. 审批委托功能

## 七、开放问题 (Open Questions)

| # | 问题 | 影响 | 建议 |
|---|------|------|------|
| Q1 | 约车付款两级审批的具体角色是什么？（如：客服主管 → 财务） | 影响审批流模板初始配置 | 需运营确认 |
| Q2 | 客户开票申请是否需要区分增值税专票和普通发票？流程是否相同？ | 影响申请表单字段设计 | 建议先支持两种，流程统一 |
| Q3 | 发票文件存储在 S3 还是本地服务器？是否需要有效期控制（签名URL）？ | 影响存储方案设计 | 建议 S3 + 签名URL，与现有 POD 文件存储保持一致 |
| Q4 | 审批委托的委托人范围：同角色 or 任意用户？ | 影响委托校验逻辑 | 建议限同角色或上级角色，避免越权 |
| Q5 | Admin 财务现有「审批」模块（库存/盘点）是否需要迁移到新引擎？ | 影响迁移工作量 | 建议 Phase 2 再评估，MVP 不动 |
