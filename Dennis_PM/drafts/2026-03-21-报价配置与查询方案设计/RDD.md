# 需求定义卡片 (RDD)

> **文档版本**: v1.0
> **创建日期**: 2026-03-21
> **阶段**: Phase 1 — 需求洞察
> **原始需求**: 页面上只能配置公司级别的报价，无法配置标准版本的报价；报价难以呈现给客户端（OMS）

---

## 1. 核心洞察 (Insight)

### 真实痛点

| # | 痛点 | 影响角色 | 业务影响 |
|---|---|---|---|
| P1 | **标准版报价无页面入口**：标准版费率（运费/附加费/仓储/操作/入库/退货等）只能通过 Excel 导入或数据库操作维护，运营无法在页面上直接创建或编辑 | 运营/销售 | 每次调价需要 IT 介入，响应周期 2-3 天；误操作风险高 |
| P2 | **客户专属报价只能只读**：Admin 后台虽有各费用 Tab 展示，但全部只读，无法直接编辑，只能通过导入 Excel 或系数调整，修改单条费率极其繁琐 | 运营/销售 | 客户议价周期长，无法快速响应客户的价格需求 |
| P3 | **客户无法自助查询报价表**：OMS 商家后台的计费页面展示的是「已发生账单」，不是「我的报价是多少」，商家需要人工询问客服才能获取报价 | 商家客户 | 客服重复查询工作量大；商家对费用感知差，不透明导致信任问题 |
| P4 | **报价版本无变更追踪**：报价修改后无日志，无法知道谁在什么时候改了什么，出现费用争议时无法溯源 | 运营/财务 | 合规风险；费用争议时无证据链 |

### JTBD（待办任务）

- **运营**：想要直接在页面上配置和管理所有费用类型的报价（运费/附加费/仓储/操作/入库/退货/包材/杂项），以便快速响应客户需求、减少 IT 依赖
- **商家**：想要随时查看自己当前生效的完整报价表，以便核对账单、评估物流成本

### 业务价值评估

| 维度 | 评估 | 说明 |
|---|---|---|
| 运营效率 | **High** | 运营报价响应速度从天级降至分钟级，减少 IT 介入 |
| 客户满意度 | **High** | 商家可自助查询，减少客服咨询量（预估降低 30%+）|
| 合规风险 | **Medium** | 变更日志解决费用争议溯源问题 |
| 营收影响 | **Medium** | 快速议价能力提升签约转化率 |

## 2. 用户故事 (User Story)

### Story 1：运营配置标准版报价

> **As a** 运营人员
> **I want to** 在 Admin 后台页面直接创建和编辑标准版费率（包含运费、附加费、仓储费、操作费、入库费、退货费、包材费、杂项费）
> **So that** 我不需要通过 IT 导入数据库，可以独立完成报价维护，响应时间从天级降到分钟级

### Story 2：运营配置客户专属报价

> **As a** 运营人员
> **I want to** 对特定客户的各类费率直接编辑（不只是只读展示），支持单条修改、批量系数调整、Excel 导入三种方式
> **So that** 我能快速给客户定制报价，并清楚看到客户报价与标准版的差异

### Story 3：商家自助查询报价

> **As a** 商家
> **I want to** 在 OMS 后台看到我当前生效的完整报价表（运费按重量×分区展示；仓储/操作费按区间展示；附加费逐项列出）
> **So that** 我可以自助核对账单费用，不需要每次联系客服确认价格

### Story 4：报价变更追溯

> **As a** 运营管理员
> **I want to** 查看每次报价修改的变更记录（谁、什么时候、改了什么、影响了哪些客户）
> **So that** 出现费用争议时有完整证据链，满足合规要求

## 3. 验收标准 & NFR

### 功能验收标准（AC）

**运营侧——报价配置**
- [ ] 运营可在 Admin 后台页面新建/编辑标准版运费费率（重量×分区），无需 Excel 导入
- [ ] 运营可在 Admin 后台页面新建/编辑标准版附加费费率（fee_code）
- [ ] 运营可在 Admin 后台页面新建/编辑标准版服务报价（仓储/操作/入库/退货/包材/杂项的 index×quote）
- [ ] 客户专属报价页各费用 Tab 支持直接编辑（Shipping/Surcharge/Handling/Storage/Receiving/Return/Misc/Material）
- [ ] 报价发布前可预览变更影响（涉及多少条费率变动、影响多少客户）
- [ ] 每次报价变更自动记录 changelog（操作人、时间、变更前后快照）
- [ ] 支持报价版本发布，发布后自动通知受影响客户

**商家侧——报价查询**
- [ ] 商家登录 OMS 后可查看当前生效的完整报价表
- [ ] 运费报价以「重量×分区」表格形式展示
- [ ] 附加费报价以「fee_code + 说明 + 金额」列表展示
- [ ] 仓储/操作/入库/退货等服务报价以「区间 + 单价 + 计费单位」列表展示
- [ ] 商家只能查看自己 company_id 下的报价（IDOR 防护）
- [ ] 支持按服务/仓库筛选
- [ ] 支持导出 PDF/Excel

### NFR（非功能性需求）

| 类别 | 要求 | 目标值 |
|---|---|---|
| 性能 | 报价查询页面加载（查快照） | < 2s |
| 性能 | 快照生成（发布后异步） | < 30s（单客户） |
| 性能 | 大批量快照生成（影响 100+ 客户） | MQ 分批，每批 < 5min |
| 数据量 | 运费费率前端分页展示 | 每页 100 条，禁止全量加载 |
| 安全 | 商家只能查自己的报价 | 后端强制用 session.company_id，禁止信任前端参数 |
| 并发 | 报价编辑乐观锁 | 提交时校验 version，冲突时提示刷新 |
| 幂等 | 快照生成并发 | 数据库唯一索引 + 事务保障，防止重复生成 |
| 保留 | 变更日志保留期 | 永久保留（软删除） |

## 4. 建议方案 (Proposal)

### MVP 方案（P0，第一阶段）

**聚焦最高频、最痛的场景：客户专属报价可编辑 + 商家报价自助查询**

1. 客户专属报价配置页各费用 Tab 从只读改为可编辑（Shipping/Surcharge/Handling/Storage/Receiving/Return/Misc/Material 共 8 个 Tab）
2. 新建变更日志表，记录所有报价修改操作
3. 新建客户报价快照机制（一主三子表），发布时异步生成
4. OMS 新增「我的报价」页，商家自助查询（查快照）

**MVP 不包含**：标准版报价页面编辑入口、版本对比、批量系数调整、导出功能

### 理想方案（P1+，后续阶段）

5. 标准版报价配置页面（新建页面，当前无入口）
6. 报价版本对比功能
7. 批量系数调整（如全部上浮 5%）
8. 报价导出 PDF/Excel
9. 订单详情页费用计算过程展示（后端数据已有，仅需前端渲染）
10. 报价版本回滚

## 5. 关键约束与依赖

| 约束 | 说明 |
|---|---|
| 数据库边界 | 所有变更只在 APP DB（app.shipsage.com）；不操作 ERP DB / WMS DB |
| 三种费率体系 | 运费（shipping_fee_rate）、附加费（fee_rate）、服务报价（service_quote）必须全部覆盖，不能只做运费 |
| 金额精度 | 所有金额字段必须 Decimal(20,6)，禁止 Float/Double |
| 租户隔离 | 所有新表含 company_id，商家查询强制用 session.company_id |
| 现有表复用 | 不新建费率存储表，复用 shipping_fee_rate / fee_rate / service_quote 系列表 |
| 快照延迟 | 快照异步生成，发布后 <30 秒生效，可接受短暂数据不一致 |

---

## 6. 专家建议

> **关于 MVP 范围**：建议第一阶段优先做「客户专属报价可编辑」+「商家自助查询」，直接解决运营和客户最痛的问题，技术风险可控。标准版报价页面编辑入口放第二阶段，避免一期范围过大。
>
> **关于快照机制**：「客户最终报价 = 标准版 + 客户差异」的合并计算是系统核心复杂度所在，建议通过「发布时异步生成快照」一次性解决，而不是每次查询时实时合并。OMS 查询性能稳定，后续做报价导出等功能也可以直接读快照，扩展性好。
